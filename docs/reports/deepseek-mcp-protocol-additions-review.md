## Summary

The DeepSeek MCP exposes richer enforcement hooks than Category 8 (Delegation Protocol) currently uses. Four gaps were identified — two are reviewer blind spots (blocking-class), two are advisory quality checks. Two additional advisory suggestions came from the reviewer's analysis of the MCP artifacts.

---

## Proposed Additions

### 1. Category 8 — Delegation Protocol (expand existing category)

**Current state:** Checks only (a) were 5+ files delegated, and (b) was output reviewed.

**Add the following checks:**

#### 1a. Plan inspection evidence *(blocking)*

The MCP server-enforces `review(agent_id, "get")` before `review(agent_id, "approve")`. The reviewer must verify Claude Code actually inspected plan content, not just called the tool. The plan exposes:

- `codebase_analysis.files_read` — files the agent examined
- `file_changes` — complete list of what will be created/modified/deleted
- `risk_assessment` — per-file risk entries

**Check:** Every file in `plan.file_changes` must appear in the git diff. Files listed in the plan but absent from the diff indicate incomplete delegation output that Claude Code should have caught during review.

**Severity:** blocking

#### 1b. Verification steps were run *(blocking)*

The DeepSeek plan includes `verification_steps` — executable behavioral commands generated by the agent to validate its own output. These are distinct from the harness's standard lint/test/build gates.

**Check:** After `poll()` returns `COMPLETED`, Claude Code must have run each command in `verification_steps`. Evidence: commands appear in the session transcript or their output appears in `output/`. If verification steps were generated but not run, the post-delegation validation gate has a hole.

**Severity:** blocking — this is the plan's own postcondition; skipping it is equivalent to skipping tests.

#### 1c. Profile selection appropriateness *(advisory)*

The MCP offers four profiles with different model/budget tradeoffs. Profile mismatches degrade output quality or waste budget.

| Task type | Required profile |
|-----------|-----------------|
| auth, security, race condition, deadlock | `deep-reason` (R1 model) |
| rename-all, migrate-all, systematic refactor | `batch-refactor` |
| bug fix, crash, small isolated fix | `quick-fix` |
| default (new features) | `default-delegation` |

**Check:** If the task touched auth/security/concurrency and `quick-fix` or `default-delegation` was used, flag as advisory.

**Severity:** advisory

#### 1d. LIMIT_REACHED means incomplete work *(blocking)*

If the agent reached its budget cap mid-execution, the git diff may show partial work. The current protocol has no guidance on this state.

**Check:** If delegation output shows the agent was in `LIMIT_REACHED` state at completion, verify the diff covers all files in `plan.file_changes`. Partial coverage = task was not completed. This is a blocking finding — do not approve incomplete delegation.

**Severity:** blocking

### 2. New Critical Nuances (append to existing list)

**Nuance 21 — LIMIT_REACHED means incomplete:**
If a delegated agent's terminal state was `LIMIT_REACHED` rather than `COMPLETED`, treat the task as incomplete regardless of what the diff shows. The plan's `file_changes` is the completion contract — any file in that list that is absent or only partially modified in the diff is a missing deliverable.

**Nuance 22 — `ask_supervisor` requires an answered question:**
If the plan or poll output shows the agent issued an `ask_supervisor` call (state: `AWAITING_TOOL_APPROVAL`, tool: `ask_supervisor`), Claude Code was required to answer the questions via `review(agent_id, "answer", answers={...})` before the agent could proceed with correct context. If no answer appears in the session, the agent made uninformed architectural choices. Treat like a missing clarification — flag as advisory.

### 3. Delegation fallback logging *(advisory — from reviewer artifact analysis)*

The `output/delegation-attempt-log.txt` artifact shows DeepSeek delegation can be blocked by security scope (the MCP's `pre_tool_use` hook or workspace confinement). When this occurs, Claude Code currently has no required behavior.

**Proposed rule:** When delegation is attempted but blocked, Claude Code must:
1. Log the attempt and reason to `output/delegation-attempt-log.txt` (already exists as a convention)
2. State explicitly in task output that delegation was attempted and why it was unavailable
3. Only proceed with direct implementation if the blocking reason is one of: security scope violation, working directory busy (`WorkingDirBusyError`), or agent mode = `claude`

**Severity:** advisory — establishes a paper trail for audit; does not block stop on its own.

---

### 4. Self-repo root as workspace *(advisory — from reviewer artifact analysis)*

The MCP's `projects_root` auto-detection adds the agent's own server repository root as an allowed workspace alongside user coding projects. This means DeepSeek could operate inside `~/.claude/Deepseek_Agent_MCP/` as a first-class workspace.

**Proposed clarification:** The protocol should explicitly state whether the MCP server's own repository (`~/.claude/Deepseek_Agent_MCP/`) is a permitted workspace root for delegation, and if so, what additional security constraints apply (e.g., never delegate tasks that modify the hooks pipeline or the MCP's own `pre_tool_use` logic).

**Recommended stance:** Allow self-repo as workspace for feature work on the MCP; prohibit delegation for any task that modifies `hooks/`, `src/core/security*`, or agent approval gate code (these touch the enforcement infrastructure itself).

**Severity:** advisory
