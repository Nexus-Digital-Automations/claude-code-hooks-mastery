## FIELD RULES (ranked by rejection frequency)

### #1 REJECTION CAUSE: tool_calls_planned
- For EVERY edit_file: old_string must be EXACT text from read_file — character-for-character
- For EVERY write_file: include the full file content in args
- For EVERY executor call: include the exact shell command
- Your supervisor WILL verify old_string against actual file contents
- Common mistakes that cause rejection:
  - Paraphrasing instead of copying exact text
  - Missing whitespace, indentation, or trailing newlines
  - Using a function name without its full context (surrounding lines)
  - Forgetting to include import statements in old_string when they're adjacent

### #2: codebase_analysis
- files_read: list EVERY file you investigated — if you plan to edit a file, it MUST appear here
- entities_found: include line numbers and exact signatures — supervisor verifies them
- Editing a file NOT in files_read is an automatic rejection

### #3: verification_steps
- At least one per major feature
- Commands must test BEHAVIOR (call the function, hit the endpoint) — not just check file existence
- Must be executable: python3, node, pytest, curl, or bash
- Supervisor executes these. You write them but do not run them.

### #4: risk_assessment
- One entry per file_change minimum
- High risk: public APIs, database schemas, auth
- Medium risk: new dependencies, shared utilities
- Low risk: new files, internal functions

### #5: steps + tools_per_step
- MUST be same length — tools_per_step[i] maps to steps[i]
- Use exact names: file_ops, executor, code_gen, git_ops, test_runner
- Order steps by dependency: models before services, services before routes

### #6: diff_preview
- SHORT pseudocode sketch per file_change (5-8 lines max)
- Key signatures, class names, config keys — NOT full file contents

## PLAN VALIDATION

Your plan is AUTOMATICALLY VALIDATED. Missing required fields = sent back for correction.

REQUIRED (plan rejected without these):
- codebase_analysis with files_read and entities_found (with signatures + line numbers)
- tool_calls_planned with EXACT args for every file operation
- verification_steps with executable behavioral commands
- risk_assessment with one entry per file_change
- file_changes with diff_preview for every entry
- steps and tools_per_step (same length)

OPTIONAL but expected for complex tasks:
- rollback_steps
- step_dependencies
